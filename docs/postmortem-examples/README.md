# Ghost Debugger — Scenario Results

This directory contains actual postmortem reports generated by Ghost Debugger
against live instrumented microservices. These are not mock outputs — they are
the result of running real failure injection scenarios and letting the AI
agent pipeline analyze actual OpenTelemetry traces, Prometheus metrics, and logs.

---

## Test Environment

- **Services:** service_a → service_b → service_c (Go HTTP microservices)
- **Instrumentation:** OpenTelemetry traces (Jaeger), Prometheus metrics, structured logs
- **Failure injection:** failure_injector service with HTTP control API
- **LLM:** Google Gemini 1.5 Flash (temperature=0)
- **Pipeline:** 7 LangGraph agent nodes with SQLite checkpointing
- **Infrastructure:** Docker Compose (Jaeger, Prometheus, ChromaDB, Grafana)

---

## Scenario 1 — Cascade Failure

**What was injected:** 2500ms artificial latency into service_b

**Expected cascade:**
- service_b becomes slow (2.5s per request)
- service_a calls service_b with a 10s timeout
- service_a's p99 latency spikes as it waits for service_b
- service_a's error rate increases when service_b calls hit deadline

**What Ghost Debugger found:**

See: `scenario1_cascade_*_postmortem.md`

**Key metrics during failure:**
- service_b p99 latency: ~2500ms (baseline: ~200ms)
- service_a p99 latency: ~2600ms (waiting for service_b)
- service_a error rate: ~15% (deadline exceeded errors)
- service_b error rate: ~0% (service_b itself not erroring, just slow)

**Ghost Debugger assessment:**
- Correctly identified service_b as the origin of the latency
- Correctly identified cascade pattern: service_b slowness → service_a timeouts
- Severity: SEV1 or SEV2 depending on traffic volume at detection time

## Scenario 2 — Resource Exhaustion

**What was injected:** 40% error rate on service_b (simulates DB connection pool exhaustion)

**Expected behavior:**
- service_b returns 500 errors on 40% of requests
- Error logs show repeated failure patterns
- service_a receives errors from service_b, propagates to callers
- Prometheus error_rate metric crosses the 5% detection threshold

**What Ghost Debugger found:**

See: `scenario2_resource_*_postmortem.md`

**Key metrics during failure:**
- service_b error_rate: ~40% (5% detection threshold)
- service_a error_rate: ~20% (secondary cascade)
- Log pattern: injected failure error messages cluster

**Ghost Debugger assessment:**
- Correctly identified service_b as the error origin
- Noted error pattern in log analysis
- Searched ChromaDB for similar past incidents

## Scenario 3 — Traffic Spike

**What was injected:** 100 rapid requests to service_a (10x normal rate)

**Expected behavior:**
- service_a request rate spikes sharply
- Downstream services (service_b, service_c) receive elevated load
- Error rate may increase as services handle burst
- Traffic spike distinguishable from service failure by request_rate pattern

**What Ghost Debugger found:**

See: `scenario3_traffic_*_postmortem.md`

**Key metrics during failure:**
- service_a request_rate: spike to 10-20 rps (baseline ~2 rps)
- Error rate: may be low if services handled the spike without errors
- Prometheus shows traffic as the anomaly, not service health

**Ghost Debugger assessment:**
- Metric agent correctly identifies request_rate spike as anomalous
- Contrast with scenarios 1/2: traffic spike, not resource failure
- Severity typically SEV2 or SEV3 depending on error rate

## Honest Limitations

1. **Log signal is weakest:** Ghost Debugger extracts logs from Jaeger span events,
   not from a dedicated log aggregation system. In production, a Loki or
   Elasticsearch integration would provide richer log analysis.

2. **Confidence depends on past incidents:** The RAG-based similar incident lookup
   provides high-confidence matches only when ChromaDB has similar past incidents.
   The seeded sample incidents cover the most common patterns.

3. **Gemini latency is the primary bottleneck:** Each LLM call takes 5-15 seconds.
   The parallel agent architecture (trace/log/metric run concurrently) reduces
   wall-clock time from ~55s sequential to ~20-30s parallel. Total pipeline
   time is dominated by the sequential phases (triage, correlation, root cause,
   postmortem writer).

4. **Detection threshold tuning required:** The incident detector's thresholds
   (5% error rate, 2000ms p99 latency) are configured for this demo environment.
   Production thresholds would be calibrated against historical baseline data.
